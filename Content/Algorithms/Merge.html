Merge

Problem Solved

Merge solves the Merge problem.

Design and Strategy

Imagine you have two lists of numbers already in order. You look at the first item in each list and pick the smaller one to place next in your combined list. Then you move forward in the list you took the item from and compare again. You keep doing this, taking the smaller of the two front items each time, until one list runs out. Finally, you append all remaining items from the other list. This simple "two‐finger" method runs in linear time and preserves sorted order.

Implementation in Java, C++, Python

Here are implementations that take two separate sorted arrays as input and output a sorted array that combined the two input arrays.

	int nA = A.length, nB = B.length;
	int[] C = new int[nA + nB];
	int i = 0, j = 0, k = 0;
	while (i < nA && j < nB) {
		if (A[i] <= B[j])

	return C;
}

Unfortunately, that version is not very useful in many contexts—especially for use in Merge Sort. The following implementation takes in a single array A together with three indices—left, mid and right—and merges the two sorted subarrays A[left…mid] and A[mid+1…right] into one sorted segment in A[left…right]. It does this by

  1. Maintaining two read pointers (i at left and j at mid+1) and a write pointer k into a temporary array,
  2. Repeatedly comparing A[i] and A[j], copying the smaller into the temp array and advancing that pointer,
  3. Once one side runs out, copying any remaining elements from the other side,
  4. And finally copying the merged result back into A[left…right].
This “in‐place” style merge integrates cleanly into the recursive structure of Merge Sort. 

public static void merge(int[] A, int left, int mid, int right) {
	int n1 = mid - left + 1;
	int n2 = right - mid;
	// allocate auxiliary arrays
	int[] L = new int[n1];
	int[] R = new int[n2];

	// copy into L and R
	for (int i = 0; i < n1; i++)  
		L[i] = A[left + i];
	for (int j = 0; j < n2; j++)  
		R[j] = A[mid + 1 + j];

	// merge back into A
	int i = 0, j = 0, k = left;
	while (i < n1 && j < n2) {
		if (L[i] <= R[j]) 
			A[k++] = L[i++];
		else              
			A[k++] = R[j++];
	}

	// copy the remaining elements of the unfinished array
	while (i < n1)  
		A[k++] = L[i++];
	while (j < n2)  
		A[k++] = R[j++];
}
void merge(int A[], int left, int mid, int right) {
	int n1 = mid - left + 1;
	int n2 = right - mid;
	// allocate auxiliary arrays
	int* L = new int[n1];
	int* R = new int[n2];

	// copy into L and R

		L[i] = A[left + i];
	for (int j = 0; j < n2; ++j)  
		R[j] = A[mid + 1 + j];

	// merge back into A
	int i = 0, j = 0, k = left;
	while (i < n1 && j < n2) {
		if (L[i] <= R[j]) 
			A[k++] = L[i++];
		else              
			A[k++] = R[j++];
	}

	// copy the remaining elements of the unfinished array
	while (i < n1)  
		A[k++] = L[i++];
	while (j < n2)  
		A[k++] = R[j++];

	// free up the dynamic memory
	delete[] L;
	delete[] R;
}
def merge(A, left, mid, right):

	L = A[left:mid+1]
	R = A[mid+1:right+1]

	i = j = 0
	k = left

			A[k] = L[i]; i += 1
		else:
			A[k] = R[j]; j += 1
		k += 1

	while i < len(L):
		A[k] = L[i]; i += 1; k += 1
	while j < len(R):
		A[k] = R[j]; j += 1; k += 1

Time/Space Analysis

Time complexity:

Although some elements may be compared multiple times—as the algorithm repeatedly checks the current element of each half—the total number of comparisons never exceeds 2n − 1, where n = right − left + 1. Each of the n elements is copied once into the auxiliary buffer and once back into the original array, so the overall work is Θ(n).

Let's be a little more detailed with the analysis. If we let n = right − left + 1 denote the total number of elements being merged, then the work breaks down into three main passes:

  1. Main merge pass: Repeatedly compare the next unused element from each half and copy the smaller into the temp buffer. This loop runs at most n times.
  2. Remainder copy pass: Once one side is exhausted, copy the remaining k elements (k ≤ n) from the other half—at most n constant-time copies.
  3. Write-back pass: Copy all n elements from the temp buffer back into A[left…right], another n assignments.

Each of these passes is linear in n. If we let c1, c2, and c3 be the per‐element costs of each pass, then T(n) = c1n + c2n + c3n = (c1 + c2 + c3)n = O(n). In actuality, there are exactly n elements copied during the first 2 passes combined, so the constants involved are slightly smaller than is suggested by this analysis. In any case, the complexity is O(n).

Space complexity: We allocate one temporary array of size n, plus a few index variables (i, j, k) in constant space. Hence the merge algorithm uses Θ(n) auxiliary space.

Summary:

  • Time complexity: Θ(n).
  • Space complexity: Θ(n).

Variations/Improvements

  • In-place merging algorithms reduce extra space at the cost of increased complexity.
  • Parallel merge can be used to divide work across processors for large datasets.

Reading Comprehension Questions

  1. Why does the merge step run in linear time?
  2. What happens when one subarray is exhausted?
  3. How much extra space does merge use?
  1. Answer: Because each element is compared at most twice and copied exactly once into the auxiliary buffer and once back, for a total of Θ(n) operations.
  2. Answer: The remaining elements from the non‐exhausted subarray are copied en masse into the merged output.
  3. Answer: Θ(n) extra space, proportional to the size of the segment being merged.
